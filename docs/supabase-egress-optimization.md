# Supabase Egress 流量优化经验总结

## 1. 问题背景 (The Problem)
在系统运行过程中，观察到 Supabase 的 **PostgREST Egress**（下行流量）异常偏高（达到 48MB），而 **Realtime Egress**（实时同步流量）相对较低（15MB）。按照单条数据 300KB 且每小时同步 6 次的估算，PostgREST 流量不应如此之高。

## 2. 根源分析 (Root Cause)
经过排查，发现流量黑洞位于 `BackupManager.tsx` 的 `fetchCloudHistory` 函数中。

*   **错误逻辑**：在获取历史版本列表时，系统使用了 `.select('id, state')`。
*   **后果**：为了在 UI 列表中显示版本的时间戳和设备名称，系统被迫下载了 **所有 20 个历史版本** 的完整 `state` 字段（即正文内容）。
*   **数据量级**：每次打开或刷新备份管理窗口，都会产生 `20 * 300KB = 6MB` 的流量消耗。频繁操作导致流量迅速飙升。

## 3. 优化方案 (The Solution)
利用 PostgreSQL 的 **JSONB 路径提取** 技术，实现“按需取词”。

*   **优化逻辑**：修改查询语句为 `.select('id, _isHistory:state->>_isHistory, _timestamp:state->>_timestamp, _device:state->>_device')`。
*   **技术原理**：Supabase 会在数据库服务器端直接提取 JSON 内部的特定字段，只返回极小的字符串片段，而不再传输整个 `state` 巨大的 JSON 块。
*   **对比效果**：
    *   **优化前**：单次列表查询 ~6MB。
    *   **优化后**：单次列表查询 ~20KB。
    *   **性能提升**：流量消耗降低了约 **300 倍**。

## 4. 稳健性评估 (Robustness Assessment)
*   **代码架构**：采用 Zustand Slice 模式，逻辑解耦清晰，无文件过长导致的读写瓶颈。
*   **持久化策略**：
    *   **15秒防抖同步**：合并频繁操作，保护上行流量。
    *   **10分钟自动备份**：平衡回溯需求与存储压力。
    *   **11分钟重叠窗口**：消除定时器延迟导致的备份丢失风险。

## 5. 流量焦虑缓解指南 (Anxiety Relief)
*   **额度对比**：Supabase 免费额度为 **5GB/月**。
*   **当前消耗**：优化后，高强度编辑 1 小时的流量消耗预计在 **5MB - 10MB** 之间。
*   **安全边际**：即使每天写作 8 小时，月度消耗也仅约 **2.4GB**，远低于 5GB 的限制。
*   **结论**：系统现在非常“吝啬”流量，除非手动执行“恢复版本”操作，否则不会产生大规模下行流量。请放心创作。
