# LensWriter 设计哲学：原子化叙事引擎

LensWriter 的核心设计理念源于对长篇小说和剧本创作的深度解构。与市面上流行的笔记软件（如 Notion）不同，LensWriter 重新定义了“块（Block）”的概念，使其真正服务于“叙事”而非“排版”。

## 1. 为什么“一段一 Block”是错误的？

Notion 引入的 Block 概念非常酷，但将其直接应用于文学创作会产生巨大的水土不服：**对于写作来说，分段是常见的排版举动，但并不是每一个段落都是一个独立的原子。**

在小说创作中，一个单独的发言或者动作描写，是没有办法承担起叙事的任务的。如果将每个段落都作为一个独立的 Block 进行管理，不仅会造成系统资源的极大浪费，更会打碎作者的叙事直觉，增加不必要的心智负担。

## 2. 叙事结构的层级划分

LensWriter 采用三级叙事结构：**Chapter (章) -> Scene (场景) -> Block (叙事单元)**。

### 2.1 Chapter (章)：完整的发布单元
*   **定义**：一个完整的故事段落，包含完整的起因、经过、结果。
*   **叙事任务**：作为连载或出版的基本单元。每一章都必须比上一章有情节和情感上的递进，必须发生矛盾并解决（至少解决一半），并且在末尾留下足够吸引人的钩子（Hook），引导读者继续阅读。

### 2.2 Scene (场景)：时空维度的切片
*   **定义**：在一个特定地点、特定时间段内发生的故事。类似于电影拍摄时，同一群演员在同一个布景下拍摄的戏份。
*   **叙事任务**：一个 Chapter 可以包含一个或多个 Scene。Scene 本身是概念性的，当我们构思一个 Scene 时，我们通常使用概括性的语言（如“主角在酒馆打听情报”），而不是具体的文字。它是承载 Block 的容器。

### 2.3 Block (块)：最小的叙事单元（原子）
*   **定义**：Block 是 LensWriter 的核心原子。它不是一个段落，而是一个**意群**或**叙事单元**。
*   **叙事任务**：例如，在一个 Scene 中，“主角A和主角B见面了，但突然闯入了第三个角色C，于是他们打了起来”。
    *   Block 1：“主角A和主角B见面了”（约400字，包含多个段落，交代见面前因和对话）。
    *   Block 2：“第三个角色C闯入”（约300字，交代闯入过程和连锁反应）。
    *   Block 3：“他们打了起来”（约1000字，详细描写心理变化和动作场面）。
*   **不可分割性**：Block 内部的几百上千字是为了完成这一个特定的叙事任务。作者可能会调整 Block 的顺序（比如倒叙：先写打斗，再写见面），但绝不需要在宏观结构上去管理 Block 内部的某一个具体段落。
*   **极简的编辑器**：这也是为什么 LensWriter 的编辑器（Tiptap）禁用了花哨的富文本功能，仅保留最基础的格式。因为过多的格式会分散作者对“叙事任务”的注意力。引入 Tiptap 仅仅是为了解决原生 textarea 在视觉上的跳动问题，提供极致平滑的输入体验。

## 3. 游离态文本的管理：Lens 与 Notes

在创作过程中，并非所有的文本都能立刻安放到具体的 Scene 中。

### 3.1 Lens (透镜)：伏笔与设定的锚点
Lens 是一种特殊的 Block。当作者写下一个伏笔的“头”时，往往已经想好了伏笔的“尾”会如何爆发。但因为还没写到具体的章节，这个伏笔只能作为一个“游离态”的 Block（即 Stashed Lens）存在。Lens 必须依附于具体的 Work（作品），它是作品内部的有机组成部分。

### 3.2 Local Notes (本地笔记 / 白色 Lens)
Local Notes 是针对特定作品的灵感碎片。根据 LensWriter 的设计哲学，这些碎片本质上也是一种“游离态的叙事单元”。它们在未来极有可能转化为正文的一部分。因此，Local Notes 应该被视为一种特殊的 Lens（白色 Lens），纳入统一的 Block 管理体系中。

### 3.3 Global Notes (全局笔记 / 收件箱)
Global Notes 独立于任何 Work 之外。它们是作者的“灵感收集箱”和备忘录。当作者产生一个想法，但还不确定它属于哪部作品，甚至不确定是否要为其开一部新作品时，这些想法会存放在 Global Notes 中。
